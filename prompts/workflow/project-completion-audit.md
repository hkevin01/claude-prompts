# Project Completion Audit

## Description
A thorough audit prompt that cross-references every promise made in README and docs against the actual codebase. Generates a test suite targeting documented behavior, produces a pass/fail completion report, and surfaces a prioritized fix list. Run this before any release.

## Prompt

```
Perform a comprehensive project completion audit for **{project_name}**.

Documentation path: `{docs_path}`
Test framework: {test_framework}

---

## PHASE 1: Documentation Inventory

Read every file in `{docs_path}` and `README.md` and compile:

1. **Feature Claims** — Every feature mentioned (e.g., "supports OAuth login")
2. **Usage Examples** — Every code example or command in the docs
3. **API/Function References** — Every public function, endpoint, or CLI flag documented
4. **Configuration Options** — Every config key mentioned with its expected behavior
5. **Performance Claims** — Any stated benchmarks, limits, or SLAs
6. **Error Behavior** — Any documented error messages or failure modes
7. **TODO/FIXME markers** — Every unresolved marker in both code and docs

Output these as a numbered inventory list.

---

## PHASE 2: Code-to-Documentation Alignment Check

For each item in the inventory:

| # | Documented Claim | Exists in Code? | Location | Status |
|---|---|---|---|---|
| 1 | Feature: OAuth login | Yes/No | src/auth.py | ✅/❌ |

Flag:
- Features documented but not implemented
- Features implemented but not documented
- Examples in docs that produce different output than the code
- Config options documented but silently ignored by the code

---

## PHASE 3: Project Structure Verification

- [ ] All files referenced in documentation exist on disk
- [ ] Directory structure matches what README describes
- [ ] All imports in source files resolve without error
- [ ] No orphaned files (present on disk but never imported/referenced)
- [ ] Dependencies in package file match what's actually imported
- [ ] Build/run command in README succeeds from a clean checkout

---

## PHASE 4: Auto-Generated Verification Test Suite

Generate a test file `tests/test_project_completion.py` (or equivalent for {test_framework}) containing:

```python
# Auto-generated by Project Completion Audit
# Tests every documented claim in README and docs/

def test_all_documented_features_exist():
    """Verify every feature mentioned in docs has an implementation."""

def test_all_readme_examples_run():
    """Run every code example from README and assert expected output."""

def test_all_documented_apis_callable():
    """Import and call every documented public function/endpoint."""

def test_all_config_options_accepted():
    """Pass each documented config key; assert no KeyError or silent ignore."""

def test_error_messages_match_docs():
    """Trigger documented error conditions; assert messages match docs."""

def test_no_broken_imports():
    """Import every module in the project; assert no ImportError."""

def test_installation_steps():
    """Run install commands from README in a subprocess; assert exit code 0."""

def test_performance_within_stated_limits():
    """Benchmark against any performance claims in docs."""
```

Each test must:
- Have a descriptive name and docstring
- Use real assertions (not just "no exception raised")
- Mock external services (APIs, databases, file I/O) cleanly
- Be independent — no shared state between tests

---

## PHASE 5: Edge Case Verification

Test each documented feature with:
- Minimal configuration (only required fields)
- Maximum documented input size
- Empty/null/zero inputs
- Invalid input types
- Missing optional dependencies

---

## PHASE 6: Completion Report

Generate `PROJECT_COMPLETION_REPORT.md`:

```markdown
# Project Completion Report — {project_name}
Generated: {date}

## Summary
- Total documented claims: X
- Verified ✅: X
- Failed ❌: X
- Missing ⚠️: X

## Feature Status
- [ ] Feature 1: ✅ PASS — verified by test_...
- [ ] Feature 2: ❌ FAIL — function missing at src/...
- [ ] Feature 3: ⚠️ PARTIAL — documented but untested

## README Claims Verification
- [ ] Example 1 (line 42): ✅ output matches
- [ ] Example 2 (line 78): ❌ produces error XYZ

## Missing / Incomplete Items
- [ ] [Critical] Feature X documented but not implemented
- [ ] [Medium] Config option Y silently ignored

## Recommendations (Priority Order)
1. [Critical] Implement Feature X (documented on README line 15)
2. [High] Fix broken example on README line 78
3. [Medium] Add test coverage for edge case Z

## Confidence Score
Overall project completion: X%
```

---

## DELIVERABLES

1. Documentation inventory (Phase 1)
2. Alignment check table (Phase 2)
3. Structure verification checklist (Phase 3)
4. Complete test suite in `tests/test_project_completion.{ext}` (Phase 4)
5. `PROJECT_COMPLETION_REPORT.md` in project root (Phase 6)
```

## Variables to Customize

- **project_name**: Used in report headers and test file comments
- **docs_path**: Folder to scan for documentation (e.g., `docs/`, `./`)
- **test_framework**: Determines test file language and syntax (`pytest`, `jest`, `vitest`, `go test`)

## Tips

- Run this before every major release
- Commit `PROJECT_COMPLETION_REPORT.md` to track quality over time
- Failures at Phase 2 (alignment) are the most valuable findings — those are promises you broke
- Use the generated test suite as a permanent regression suite, not a one-off

## Engineering Rigor Standard

Use this checklist when executing the prompt. Treat every unchecked item as a delivery blocker.

### Requirements Quality

- Define explicit acceptance criteria before implementation.
- Capture assumptions, constraints, and out-of-scope items.
- Maintain bidirectional traceability: requirement -> design -> code -> test.
- Reject ambiguous requirements; request clarification or record assumptions.

### Design Quality

- Document architecture decisions and rejected alternatives.
- Specify component interfaces, contracts, and failure behaviors.
- Define data ownership, lifecycle, and schema evolution strategy.
- Identify safety, security, reliability, and performance risks up front.

### Implementation Quality

- Enforce deterministic behavior and idempotency for repeatable operations.
- Validate all external inputs at boundaries with explicit error messages.
- Use structured error handling with categorized failure codes.
- Avoid hidden global state and implicit side effects.
- Keep functions cohesive, testable, and bounded in responsibility.

### Verification & Validation

- Implement tests for happy path, edge cases, and failure modes.
- Add integration tests for cross-component contracts.
- Add regression tests for each bug fix.
- Verify non-functional requirements: latency, throughput, memory, and resilience.
- Record evidence of verification (commands, outputs, metrics, and artifacts).

### Security & Compliance

- Apply least privilege for credentials, services, and runtime permissions.
- Never hardcode secrets; use secure env/config mechanisms.
- Validate authn/authz paths and enforce auditability for sensitive actions.
- Include dependency and supply-chain checks in the workflow.

### SDLC Execution Gates

A phase is complete only when all gates pass:

- [ ] Scope gate: requirements are complete, versioned, and traceable.
- [ ] Build gate: build is reproducible from a clean checkout.
- [ ] Test gate: required test suites pass with evidence.
- [ ] Quality gate: lint/static checks pass; no critical warnings unresolved.
- [ ] Security gate: critical/high vulnerabilities addressed or risk-accepted in writing.
- [ ] Documentation gate: README, runbooks, and change notes updated.
- [ ] Release gate: rollback plan, monitoring hooks, and ownership are defined.

### Output Requirements

Every response generated from this prompt should include:

1. Objective and assumptions
2. Phased execution plan with gates
3. Implementation details and impacted files/components
4. Verification evidence and residual risk summary
5. Follow-up backlog prioritized by impact and risk
